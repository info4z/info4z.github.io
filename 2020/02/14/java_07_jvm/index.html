jvm性能篇

一、类加载机制

JVM用来存储加载的类信息、常量、静态变量、编译后的代码等数据。

虚拟机规范中这是一个逻辑区划。具体实现根据不同虚拟机来实现。

Oracle 的 HotSpot 在 java7 中方法区放在永久代，java8放在元数据空间，并且通过GC机制对这个区域进行管理。

（二）类加载器

类加载器负责装入类，搜索网络、jar、zip、文件夹、二进制数据、内存等指定位置的资源。

一个 java 程序运行，最少有三个类加载器实例，负责不同类的加载。

  • Bootstrap Loader：核心类库加载器，C/C++ 实现，无对应 java 类：null。加载 JRE_HOME/jre/lib 目录或者用户配置的目录。JDK 核心类库：rt.jar（核心运行时库），String 和 System 就在其中定义。
  • Extension Class Loader：扩展类库加载器，ExtClassLoader 的实例。加载 JRE_HOME/jre/lib/ext 目录、JDK 扩展包、或者用户配置的目录。
  • Application Class Loader：用户应用程序加载器，AppClassLoader 的实例。加载 java.class.path 指定的目录，用户应用程序 class-path 或者 java 命令运行时参数 -cp 。

3、类不会重复加载

类的唯一性：同一个类加载器，类名一样，代表是同一个类。

识别方式：ClassLoader Instance Id + PackaeName + ClassName
验证方式：使用类加载器，对同一个 class 类的不同版本，进行多次加载，检查是否会加载到最新的代码

验证：程序运行中修改 HelloService 中代码，经重新编译后输出内容无变化，说明 .class 文件不会被重复加载

5、类的卸载

类什么时候会被卸载？满足如下两个条件：

  1. 该 Class 所有的实例都已经被 GC
  2. 加载该类的 ClassLoader 实例已经被 GC

验证：jvm 启动中增加 -verbose:class 参数，输出类加载和卸载的日志信息

5、双亲委派模型

类加载器之间不存在父类子类的关系，“双亲”是翻译，可以理解为逻辑上定义的上下级关系。

graph LR
	A(Bootstrap Loader<br>顶层类加载器)
	B(Extension Class Loader<br>扩展类库加载器)
	C(Application Class Loader<br>用户应用程序加载器)
	D(最底端子类加载器)
	A --> |查找|B --> |查找|C --> |查找|D --> |委托|C --> |委托|B --> |委托|A

为了避免重复加载，由下到上逐级委托，由上到下逐级查找

  • 首先不会自己去尝试加载类，而是把这个请求委托给父加载器去完成。每一个层次的加载器都是如此，因此所有的类加载请求都会传给上层的启动类加载器。
  • 只有当父加载器反馈自己无法完成该加载请求（该加载器的搜索范围中没有找到对应的类）时，子加载器才会尝试自己去加载。

二、垃圾回收机制

C 语言中，分配和释放内存是一个手动过程。
在 java 中，解除内存分配的过程由垃圾收集器自动处理。

（一）自动垃圾收集

自动垃圾收集是查看堆内存，识别哪些对象正在使用、哪些对象未被删除、删除未使用对象的过程。

使用中的对象或引用的对象意味着程序的某些部分仍然维护指向该对象的指针。程序的任何部分都不再引用未使用的对象或未引用的对象，因此可以回收未引用对象使用的内存。

思考：如何确定内存需要被回收？

第一步称为标记。这是垃圾收集器识别哪些内存正在使用而哪些不再使用的地方。

不同类型内存的判断方法：

  • 对象回收：可达性算法

    引用计数也是一种常见的判断方法，简单来说就是引用一次就加1，用完之后就减1。java中很少用，会造成循环引用的问题。

  • 方法区回收

（二）可达性分析算法

简单来说，将对象及其引用关系看作一个图。选定活动的对象作为 GC Roots，然后跟踪引用链条，如果一个对象和 GC Roots 之间不可达，也就是不存在引用，那么即可认为是可回收对象。

核心结论：从 GC Roots 出发，沿引用链遍历，能到达的对象存活，不可达的就是垃圾。

那么，什么是活动的对象？或者说谁可以作为 GC Root 对象？

  • 虚拟机栈中正在引用的对象
  • 本地方法栈中正在引用的对象
  • 静态属性引用的对象
  • 方法区常量引用的对象

那，GC Root 是怎么引用活动对象的？

  • StrongReference：强引用。最普通的引用，Object obj = new Object() 就是强引用。
    只要强引用还在，GC 绝不回收该对象。 它是 GC 判断对象是否存活的唯一标准。

  • SoftReference：软引用。用 SoftReference<T> 包装。内存充足时，软引用对象不会被回收。只有 JVM 内存紧张（接近 OOM）时，才会回收软引用对象。

    适合做内存敏感型缓存，内存够就留着，不够就牺牲。

  • WeakReference：弱引用。用 WeakReference<T> 包装。不管内存够不够，下次 GC 一定回收。弱引用对象的生命周期很短，只要发生 GC 就会被清理。

    典型场景：WeakHashMap，key 是弱引用，GC 后自动删条目，防止内存泄漏

  • PhantomReference：虚引用。用 PhantomReference<T> 包装，必须配合 ReferenceQueue 使用。get() 永远返回 null，无法通过虚引用拿到对象。对象被回收后，虚引用会被放入 ReferenceQueue。

    唯一目的：在对象被回收（finalized）后，得到一个通知，做最后的资源清理（如释放堆外内存）

最后，怎么判断可达性？

  • Strongly Reachable：强可达。一个对象可以有一个或多个线程可以不通过各种引用访问到的情况。
  1. 软可达(Softly Reachable)：就是当我们只能通过软引用才能访问到对象的状态
  2. 弱可达(Weakly Reachable)：只能通过弱引用访问时的状态; 当弱引用被清除的时候，就符合销毁条件
  3. 幻象可达(Phantom Reachable)：不存在其他引用，并且 finalize 过了，只有幻象引用指向这个对象
  4. 不可达(unreachable)：意味着对象可以被清楚了

总结：强引用不收，软引用缺内存才收，弱引用GC必收，虚引用收完通知你。

（三）垃圾收集算法

标记-清除（Mark-Sweep）算法

  • 首先标识出所有要回收的对象，然后进行清除；
  • 标记清除过程效率有限，有内存碎片化问题，不适合特别大的堆；
  • 垃圾收集算法基本基于“标记-清除”的思路进行改进。

复制（Copying）算法

  • 划分两块同等大小的区域，收集时将活着的对象复制到另一块区域，原区域清空；
  • 拷贝过程中将对象顺序放置，就可以避免内存碎片化；
  • 复制 + 预留内存，有一定的浪费。

标记-整理（Mark-Compact）

  • 类似于“标记-清除”，但为避免内存碎片化，它会在清理过程中移动对象位置，以确保移动后的对象占用连续的内存空间；
  • 避免内存碎片化。

（四）分代收集

根据对象的存活周期，将内存划分为几个区域（Eden，form，to，Tenured），不同区域采用不同的垃圾收集算法。

新对象会分配到 Eden（新生代），可以设置-XX:+PrentenureSizeThreshold: 如果超过阈值，大对象直接进入老年代（Tenured）。

新生代采用复制算法：Eden 中的对象经过一次 GC 后，如果未回收则 Copy 进入 form。第二次 GC 后，将 Eden 和 form 中的对象统一 Copy 到 to。第三次 GC 后，将 Eden 和 to 中的对象 Copy 到 form 中。GC 一定次数未回收的对象移动到老年代 (-XX:MaxTenuringThreshold)。

老年代采用标记-整理，也就是先标记，然后在清除过程中移动对象，确保占用的内存是连续的。

（五）垃圾收集器

1、串行收集器

Serial GC：-XX:+UseSerialGC
Serial Old GC：-XX:+UseSerialOldGC

Client 模式下 JVM 的默认选项。单个线程来执行所有垃圾收集工作，适合单处理器设备。可以在老年代使用，它采用了标记整理（Mark-Compact）算法，区别于新生代的复制算法。

如图所示：棕色线条表示 GC 线程，它是启动一个线程来进行标记、清除、整理。当 GC 在工作的时候，有一个很重要的特性，叫“stop the world”，简称 STW。在这段时间里，你的程序对外表现为”完全卡死”——请求不响应、接口超时、游戏掉帧。

垃圾回收时，JVM 暂停所有应用线程，只让 GC 线程运行，直到回收完成。

2、并行收集器

Parallel GC：-XX:+UseParallelGC
Parallel Old GC：-XX:+UseParallelOldGC

Server 模式 JVM 的默认 GC 选项，整体算法和 Serial 比较相似，区别是新生代和老年代 GC 都是并行进行。可以设置 GC 时间或吞吐量等值，可以自动进行适应性调整 Eden，Survivor 大小和 MaxTenuringThreshold 的值。也称为吞吐量优先的 GC。吞吐量为代码运行时间在总耗时（代码运行时间与GC停顿时间总和）中的占比。
$$
吞吐量 = \frac{用户代码运行时间}{用户代码运行时间 + GC停顿时间}
$$

3、并发收集器——专用于老年代

CMS（Concurrent Mark Sweep）GC：-XX:+UseConcMarkSweepGC

专用老年代，基于标记清除（Mark-Sweep）算法，设计目标是尽量减少停顿时间。采用的标记清除算法，存在着内存碎片化问题，长时间运行等情况下发生 full GC（不区分新生代老年代，直接全局 GC）导致恶劣的停顿。

CMS 会占用更多的 CPU 资源，并和用户线程争抢。减少了停顿时间，这一点对于互联网 web 等对时间敏感的系统非常重要，一直到今天，仍然有很多系统使用 CMS GC。

4、其他

1）ParNew GC：并行收集器

-XX:+UseParNewGC

新生代 GC 实现，它实际是 Serial GC 的多线程版本。可以控制线程数量，参数：-XX:ParallelGCThreads。

最常见的应用场景是配合老年代的 CMS GC 工作，参数：-XX:=UseConcMarkSweepGC。

2）G1 GC：并发收集器

-XX:+UseG1GC

针对大堆内存设计的收集器，兼顾吞吐量和停顿时间，JDK9 后默认选型，目标是替代 CMS。

G1 将堆分成固定大小的 region（区域），各区域间是复制算法，但整体上实际可看作是“标记-整理”算法，可以有效地避免内存碎片。

如图所示：红色新生代（Eden和Surivor），淡蓝色老年代。找不到大内存时执行FullGC。

（六）垃圾收集器组合

常用选择

  1. 默认选择: Parallel Scavenge + Parallel Old
  2. 常用选择: ParNew + CMS
  3. 激进选择: G1

三、JDK内置命令工具

官方文档：https://docs.oracle.com/javase/8/docs/technotes/tools/

（一）基本工具

javap：反编译工具，主要用于根据 java 字节码文件汇编为 java 源代码文件

（二）监控工具

jps：显示当前所有 java 进程 pid 的命令

jstat：监视 java 虚拟机（JVM）统计信息

Jconsole：java 监控和管理控制台。在测试和性能调试的时候可以用的可视化工具。

JvisualVM：虚拟机工具。比 Jconsole 的要强大很多，不仅可以监控远程，主要是可以分析内存快照。

（三）故障排除工具

jcmd：发送诊断指令请求到运行的 JVM ，可以替代 jps 工具查看本地的 jvm 信息

jinfo：生成配置信息，可以查看运行中 jvm 的全部参数，还可以设置部分参数

jmap：打印出 java 进程内存中 Object 的情况，或者将 VM 中的堆，以二级制输出成文本

jhat：分析 java 堆的命令，可以将堆中的对象以 html 的形式显示出来，支持对象查询语言 OQL

jstack：堆栈跟踪工具。

jstack 用于打印出给定的 java 进程 ID 或 core file 或远程调试服务的 java 堆栈信息，如果是在 64 位机器上，需要指定选项 “-J-d64”
